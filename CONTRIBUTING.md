# Contributing

Thanks for your interest in contributing. This toolkit exists because FedRAMP
20x is moving fast and CSPs need a reference implementation that stays current.
Every contribution helps the community keep up.

## Ground rules

1. **The capability is the source of truth.** Never hardcode framework text
   into a renderer or a Terraform module. Reference IDs (`KSI-IAM-01`,
   `IA-2(1)`) and let the loader resolve them at render time.
2. **Authored vs. synced.** Anything under `frameworks/` is generated by the
   sync workflow and must never be hand-edited. Anything under `capabilities/`,
   `evidence/`, `policies/`, `terraform/`, or `renderers/` is authored.
3. **Cite provenance.** Every capability ships with a `provenance` block.
   Where possible, name the assessment that validated the pattern (CSP +
   3PAO + date + public URL).
4. **No vendor secrets in evidence collectors.** Use environment variables
   for every credential. The collectors must be safe to share publicly.

## What kind of contribution

| Type | What to do | What to expect |
|---|---|---|
| Add a capability | Copy `capabilities/iam/mfa-phishing-resistant.yaml` as a template, edit the statement and mappings, link evidence + Terraform | Schema + drift CI must pass; one maintainer review |
| Fix a mapping | Edit the affected capability's `satisfies:` block, update `provenance.last_reviewed` | Drift CI must pass |
| Add an evidence collector | Drop a script under `evidence/<source>/`, reference it from a capability's `evidence` array | Must respect `set -euo pipefail`, fail loudly on missing env vars |
| Add a renderer | New file under `renderers/<framework>.py`, must consume the loader API only | One maintainer review + a sample artifact in the PR |
| Add a framework | Extend `_schema.yaml` with the new `satisfies:` key, write a sync adapter under `sync/`, write a renderer | Two-maintainer review since this changes the schema |

## Local development

```bash
git clone <repo-url>
cd grc-toolkit
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

# Validate schema + index
python -m renderers.shared.capability_loader

# Run all tests
pytest -v

# Generate sample outputs
python -m renderers.rev5_ssp --out samples/ssp.docx
python -m renderers.fedramp_20x --out samples/20x.json
```

## PR checklist

- [ ] `pytest` passes locally
- [ ] `python -m renderers.shared.capability_loader` runs clean
- [ ] If you added/edited a capability, `provenance.last_reviewed` is today
- [ ] If you added evidence collectors, they fail safely without credentials
- [ ] No framework text copied verbatim into authored content (use IDs)
- [ ] No personal data, real account IDs, or secrets in the diff

## Adding a capability — worked example

Say you want to add a capability for "All data encrypted at rest with
customer-managed KMS keys."

1. Pick a slug: `cap-data-at-rest-cmk`.
2. Create `capabilities/crypto/data-at-rest-cmk.yaml`. Use the schema in
   `capabilities/_schema.yaml` as your guide.
3. Write the `capability_statement` in plain English. No framework jargon.
4. Map to frameworks:
   ```yaml
   satisfies:
     fedramp_20x:
       - ksi: KSI-SVC-04   # cryptographic key management
         coverage: full
     fedramp_rev5:
       - control: SC-13
         coverage: full
       - control: SC-28
         coverage: full
       - control: SC-28(1)
         coverage: full
   ```
5. Add evidence collectors under `evidence/aws/`, `evidence/gcp/`, etc., and
   list them in the `evidence` block.
6. Add Terraform module pointers under `terraform/<cloud>/modules/...`.
7. Run `pytest` — drift tests will tell you immediately if any of the IDs
   you referenced no longer exist upstream.
8. Open a PR. Maintainer reviews the statement for accuracy and the mappings
   for completeness.

## Reporting a stale mapping

If FedRAMP changes a KSI definition and a capability's `coverage` field is
no longer accurate, open an issue with:

- Affected capability ID
- KSI/control ID
- Link to the FedRAMP change (RFC or FRMR release)
- Suggested new coverage value

The drift watcher should catch most of these automatically, but human
review for `partial`-vs-`full` calls is welcome.

## Code of conduct

Be kind. Compliance is hard enough.
